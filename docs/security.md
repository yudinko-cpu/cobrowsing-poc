# Безопасность

Документ описывает фактическое состояние безопасности PoC-стенда: что защищено
и как, модель угроз, обращение с данными. Аудитория — ИБ и техлиды.
Архитектура — в [architecture.md](architecture.md), стенд —
в [test-stand.md](test-stand.md).

Важная рамка: это **PoC**. Решения принимались с приоритетом скорости проверки
гипотезы; известные ограничения перечислены явно в §4, а не спрятаны.

## 1. Реализованные меры защиты

### 1.1 Транспортное шифрование

- Весь HTTP/WS-трафик (API, дашборд, LiveKit signaling) — только HTTPS/WSS.
  TLS-терминация в Caddy, сертификаты Let's Encrypt с авто-продлением.
- WebRTC-медиа шифруется на уровне протокола: DTLS-handshake + SRTP. Медиа
  идёт напрямую клиент ↔ SFU и не расшифровывается нигде, кроме конечных
  точек и самого SFU (терминирует SRTP по определению архитектуры).
- Внутренние соединения (Caddy → backend/web-agent/LiveKit, backend → Redis)
  идут по localhost / изолированной bridge-сети одного хоста и не покидают VPS.

### 1.2 Аутентификация и авторизация сессий

- **JWT-токены LiveKit** подписываются секретом (HS256, секрет ≥32 символов,
  генерируется `openssl rand -base64 32`). Токен ограничен: конкретная комната
  (`roomJoin` + `room`), identity участника, TTL 1 час. Токен на комнату A не
  даёт доступа к комнате B.
- **Клиент и агент получают разные токены** со своими identity; выдача — только
  через backend-эндпоинты, LiveKit API-ключи клиентам не выдаются никогда.
- **6-значный код сессии** — единственный способ агента попасть в комнату:
  - генерируется через `crypto.randomInt` (CSPRNG, без modulo bias);
  - TTL 10 минут на ввод;
  - **anti-hijack claim**: первый `agent/join` фиксирует `agentId` за кодом,
    попытка входа с другим `agentId` → 409. Повторный вход того же агента
    (F5, ретраи) — переиздание токена с той же identity;
  - имя комнаты — `cobrowse-<uuid4>`, неугадываемое: знание кода не раскрывает
    комнату, знание комнаты не даёт токена.
- **Rate limiting**: 30 запросов/мин с IP на все эндпоинты backend
  (express-rate-limit) — защита от перебора кодов. Комбинаторика: 10⁶ кодов,
  живущих ≤10 минут, при 30 попытках/мин с IP перебор нереалистичен без
  распределённой атаки.
- **CORS** — явный allowlist origins (в проде — только домен дашборда),
  функциональный matcher с логированием отклонённых origins.

### 1.3 Согласие пользователя

- Стрим **не может начаться без явного consent** — prompt на стороне iOS до
  любого захвата экрана; отказ оставляет state machine в `idle`.
- Захват — **in-app ReplayKit**: транслируется только экран самого приложения.
  При сворачивании приложения системные экраны и другие приложения в поток
  не попадают.
- Индикатор записи (REC) виден пользователю всю сессию; клиент может завершить
  сессию в любой момент, `session/end` доступен обеим сторонам.

### 1.4 Сетевой периметр и хост

- Firewall (ufw + Hetzner Cloud Firewall): наружу открыты только 22, 80, 443,
  7881/tcp и 50000–60000/udp. Проверено сканированием извне.
- Redis не слушает внешние интерфейсы (только 127.0.0.1 и bridge-сеть);
  backend и web-agent опубликованы только на loopback, доступ — через Caddy.
- SSH — только по ключам; деплой от отдельного непривилегированного
  пользователя `deploy` (не root), ключ деплоя существует только в GitHub
  Secrets и `authorized_keys`.
- Docker-образы — приватные в GHCR; секреты стенда — в `.env` на VPS с
  доступом только у `deploy`.

### 1.5 Наблюдаемость

- Структурированные JSON-логи backend (pino): создание сессий, join'ы с
  agentId, отклонённые CORS-origins, anti-hijack события, завершения.
- Access-логи Caddy по каждому домену (JSON, файлы в volume).
- Fail-fast конфигурация: сервис не стартует с неполным конфигом — исключает
  «тихую» работу с дефолтными секретами.

## 2. Модель угроз

Актив, который защищаем: **видеопоток экрана пользователя** (потенциально
содержит PII, финансовые данные) и доступ к живой сессии.

| # | Угроза | Вектор | Митигация | Остаточный риск |
|---|---|---|---|---|
| T1 | Перехват трафика (MitM) | Публичные сети клиента/агента | TLS везде; SRTP/DTLS для медиа | Низкий |
| T2 | Подбор кода сессии | Брутфорс `/agent/join` | 10⁶ кодов × TTL 10 мин × rate limit 30/мин/IP; логирование | Низкий; распределённый перебор заметен в логах, авто-бана нет |
| T3 | Угон занятой сессии | Ввод чужого валидного кода | Claim по agentId, 409 чужому агенту | Средний: agentId самодекларируемый (нет аутентификации операторов) — см. §4 |
| T4 | Несанкционированный просмотр экрана без ведома клиента | Компрометация SDK / скрытый запуск | Consent prompt обязателен, REC-индикатор, in-app capture | Низкий в рамках модели «SDK доверен приложению-хозяину» |
| T5 | Доступ к инфраструктуре | SSH, открытые порты, Redis | Только ключи, non-root deploy, firewall, Redis на loopback | Средний: нет fail2ban, авто-обновлений, mTLS |
| T6 | Кража/утечка JWT | Логи, история браузера, пересылка | Токен короткоживущий, привязан к комнате и identity | Средний: TTL 1 час — долго для PoC-прода |
| T7 | Компрометация CI/CD | Утечка GHA-секретов, supply chain образов | SSH-ключ только в Secrets, деплой-юзер ограничен, образы приватные | Средний: нет подписи образов, нет pin'а базовых образов по digest |
| T8 | DoS на media-порты | Флуд UDP 50000–60000 | Ничего специального (ICE consent freshness на уровне WebRTC) | Принят для PoC |
| T9 | Просмотр экрана «лишним» участником | Подключение третьей стороны в комнату | Токены только через backend, комната ≤5 участников, неугадываемое имя комнаты | Низкий |

Доверенные компоненты (в границах модели): приложение-хозяин SDK, VPS и его
администратор, GitHub-репозиторий и Actions. Оператор считается
**полудоверенным** — сейчас его сдерживает только знание кода (см. T3).

## 3. Данные и приватность

### 3.1 Какие данные проходят через систему

| Категория | Что именно | Где находится | Хранение |
|---|---|---|---|
| Видеопоток экрана | Всё содержимое экрана приложения — потенциально PII, платёжные данные, документы | Транзитом через LiveKit SFU (RAM) | **Не записывается и не сохраняется** — Egress не развёрнут |
| Аудио | Голос клиента и оператора | Транзитом через SFU | Не записывается |
| Метаданные сессии | Код, roomName, customerId (опционален, по умолчанию `anon-*`), agentId, таймстемпы, статус | Redis | TTL: код 10 мин→1 ч, сессия 1 ч; персистентность RDB (snapshot на диск VPS) |
| Логи | session-события, IP в access-логах Caddy, agentId | Файлы/volumes на VPS | Без ротации и политики удаления (PoC) |

### 3.2 Ключевые свойства

- **Ничего не уходит третьим сторонам**: весь путь данных — устройство клиента
  → наш VPS → браузер оператора. Cloudflare — только DNS (без проксирования),
  внешних аналитик/трекеров нет.
- **Запись сессий отсутствует** по умолчанию и физически (Egress-контейнер не
  развёрнут). Включение записи — отдельное решение с отдельными требованиями
  (хранение, шифрование at-rest, retention, правовые основания).
- **Минимизация данных**: backend не знает ничего о пользователе, кроме
  опционального `customerId`, который задаёт приложение-хозяин; по умолчанию
  используется анонимный идентификатор.
- **Оператор видит экран целиком.** Redaction (маскирование полей с PII) в
  video-streaming-подходе не реализован — это осознанный архитектурный
  trade-off ([architecture.md, §6.4](architecture.md)). Для пилота в
  финсекторе это главный вопрос к проработке: варианты — маскирование на
  стороне SDK до кодирования кадра (blur областей по координатам «приватных»
  view) либо смена подхода на scene-graph.
- Согласие пользователя — явное, до начала передачи, с видимым индикатором
  на протяжении всей сессии (§1.3).

## 4. Известные ограничения PoC

Зафиксировано как принятые риски скоупа PoC, не как упущения:

1. **Нет аутентификации операторов** — дашборд открыт, agentId
   самодекларируемый. Для пилота: SSO/OIDC перед дашбордом, agentId из
   токена, RBAC.
2. **Нет redaction PII** в видеопотоке (см. §3.2).
3. **Секреты в `.env` на диске** VPS, ручная синхронизация LiveKit-ключа,
   без ротации. Для пилота: SOPS/Vault, ротация ключей.
4. **Нет E2EE медиа** — SFU видит расшифрованный поток (это свойство любой
   SFU-архитектуры). LiveKit поддерживает E2EE поверх — потребует доработки
   клиентов.
5. **Хост-гигиена**: нет fail2ban, unattended-upgrades, подписи образов,
   ротации логов, мониторинга безопасности.
6. **Логи содержат IP и agentId** без политики retention.
7. **JWT TTL 1 час** без механизма отзыва (отзыв возможен только через
   deleteRoom).
